01SELF-INFLICTED SIGTERM

The "crash loop" isn't a crash loop.

src/persistent-process.ts · replace per-turn killer with session-level recycling
30 min

What's actually happening

Atlas runs a persistent Claude CLI process so it doesn't pay the 2-5 second cold-start cost on every Telegram message. Smart. But somewhere along the way, a "heavy turn" killer got added: any reply that uses ≥5 tools OR takes ≥120 seconds triggers a full process tear-down and respawn.

The intent is real — the code comment notes "the CLI accumulates internal state from tool calls; after enough complexity, the next message hangs." So this isn't a safety mechanism, it's a recycling mechanism aimed at a state-leak bug. The problem is the threshold: modern Opus replies routinely cross both 5 tools and 120 seconds on normal work. A "draft and send the newsletter" turn might fire 12+ tools over 4 minutes. The killer doesn't catch runaway turns — it catches basically every meaningful turn.

Meanwhile Atlas already has a per-turn watchdog at src/persistent-process.ts:856 that's the real safety net: for Opus, 540s of inactivity or 2,700s wall clock (45 min) ends a stuck turn. That's the layer protecting against genuine hangs. The heavy-turn killer fires 22 minutes before the watchdog even gets warmed up.

Today's log shows 10 self-kills, every one triggered by a normal Opus reply (8 tools / 65s, 27 tools / 307s, 15 tools / 460s). The parent Atlas process itself has been up 18+ hours without a real crash.

How Hermes solves the same problem

Hermes (Nous Research's open-source agent, running in your ~/.hermes/profiles/derek) doesn't use a per-turn killer at all. Its config uses three knobs:

max_turns: 90  — conversation-session cap, not per-turn
gateway_timeout: 1800s  — 30 min inactivity; resets on every tool call
context_compression.threshold: 0.50  — compress middle turns at 50% of model context

The philosophy from the Hermes config comments: "The agent can run indefinitely as long as it's actively calling tools or receiving API responses."

Hermes can compress context to handle long sessions because it runs in-process Python. Atlas runs the Claude CLI as a subprocess, so the equivalent move is to recycle the whole process every N turns via --resume. Same goal, different mechanism. The fix below is "Hermes's max_turns shape, translated for a CLI-subprocess world."

02DOUBLE-SPAWN RACE

Every restart spawns two replacements.

src/persistent-process.ts · the restart()  /  monitorExit() race · three-line fix
20 min

What's actually happening

When restart() runs, it SIGTERMs the old Claude CLI and then directly calls spawnProcess() to start a new one. Reasonable.

But the dying process still has a monitorExit() handler attached to its proc.exited promise. When the kill lands, that handler fires too. It checks if (this.status === "shutdown") return; — but the status is still "alive", because restart() hasn't gotten around to flipping it yet. So monitorExit() proceeds, calls autoRestart(), and spawns another Claude CLI.

Result: two children per restart, ~2 seconds apart. The orphan eats memory and an OAuth slot until its idle timer eventually fires. Once Fix 01 lands and restart frequency drops from ~10/day to ~1/day, this matters less — but the race is still real and the fix is trivial.

The key insight: monitorExit() already has a status-based bail pattern (it returns early on "shutdown"). The ProcessStatus enum at line 49 already includes "restarting" as a value (used by autoRestart()). We just need to set that status earlier in restart() and recognize it in monitorExit(). No new fields. No new flags.

How Hermes avoids the problem

Hermes is in-process Python on a single asyncio event loop — it never spawns a separate Claude CLI subprocess, so there's no death-monitor that can race with a deliberate kill. The closest analog is the gateway agent restart, which uses an explicit drain pattern from config.py:

restart_drain_timeout: 180s  — stop accepting new work, wait for in-flight agents, interrupt stragglers

One coordinator, one event loop, no race. Atlas can't replicate this directly because the Claude CLI is a separate OS process with its own lifecycle — the supervisor has to coordinate with a stranger across a process boundary. The status-flag fix is the cross-process equivalent of Hermes's "stop accepting new work" signal.

03SILENT KNOWLEDGE LOSS

Your ingest pipeline is throwing away parsed chunks.

src/reader.ts · stricter about malicious, looser about messy
20 min

What's actually happening

Reader.ts isn't just a parser — it's a security boundary. The file header describes a CaMeL Reader pattern: untrusted content (emails, PDFs, web ingest, CRM messages) goes through a Haiku call with no tool access, gets extracted to a typed schema, and only the typed extraction reaches the tool-enabled Planner. This prevents prompt-injection payloads in ingested content from triggering real actions.

Two safety layers protect that boundary: JSON.parse must succeed, and a strict schema check (line 79-83) rejects any unknown key. When either fails, gateChunk returns null and drops the chunk. That fail-closed behavior is correct.

The bug isn't that we drop chunks. The bug is that we drop chunks when Haiku wraps perfectly-valid JSON in markdown fences (```` ```json {...} ``` ````) or adds a "Sure, here's the JSON:" preamble. The current code does a bare JSON.parse(result.text) — no fence-stripping — so the parse throws on every fenced response. Today alone you lost chunks from journals, the PV knowledge layer, HRTU course materials, and Telegram ingest. Real knowledge missing from real prompts.

The fix makes the parser stricter about what's malicious and looser about what's just messy. Fail-closed posture stays intact for everything actually suspicious.

How Hermes handles structured output

Hermes solves this at the framework layer using DSPy Signatures — typed input/output declarations that auto-generate format instructions and parse responses with multiple fallback strategies (JSON, key-value, completion-style). When parses fail at scale, GEPA reads the failure traces and mutates the prompt based on what actually went wrong:

DSPy Signature  — typed schema with auto-generated format prompt
GEPA reflection  — reads parse-failure traces, mutates prompt to be more robust
Pareto frontier  — keeps multiple working prompt variants instead of converging

A Hermes-equivalent system would learn to add a few-shot example automatically after seeing N parse failures. Atlas doesn't have DSPy or GEPA, so we're doing manually what Hermes does automatically: tighter prompt with an explicit example shape, plus a lenient parser, plus the fail-closed safety net. Smaller scale, same playbook.

04SEQUENTIAL HAIKU LOOP

Context search is 8× slower than it needs to be.

src/search.ts · combine + parallelize the two CaMeL gate loops
10 min

What's actually happening

When Atlas pulls relevant context for a turn, it reranks chunks across two untrusted sources (documents and maa_knowledge) and runs each through the CaMeL gateChunk(). Each gate is a Haiku CLI subprocess spawn — about 3-4 seconds of CLI startup per call.

There are two sequential bugs in the current code at search.ts:300-322:

  1. Each loop is itself sequential — every chunk awaits the previous one. 8 chunks × ~3.5s = ~28s.
  2. The two loops run sequentially relative to each other — documents finish before maa_knowledge starts.

Worst-case wall clock today: (N_docs + N_maa) × 3.5s. With the 12s deadline at relay.ts:4084, even 4 chunks blow it. The timeout returns an empty string and Atlas proceeds without the knowledge it was supposed to have. Silent quality degradation every time rerank surfaces enough chunks to matter.

Is parallelizing 8 Haiku CLIs safe? (yes — analysis)

The obvious worry: 8 concurrent CLI subprocesses could hit OS limits, Anthropic rate limits, or saturate the Max-plan OAuth token. Walked through it; none of these bind at this scale.

CONCERN REALITY AT 8 CONCURRENT
OS subprocess limitWindows handles hundreds. 8 = noise.
Memory (each CLI ≈ 80MB)640MB peak vs 16-32GB available.
Anthropic rate limitsMax plan well above 8/s. Retry logic at haiku-client.ts:77 absorbs 429/529.
OAuth token concurrencyNo file lock on read. Server caps absorbed by per-call retry.
CLI startup amplificationOS file cache shares cost across spawns 2-8. Expect ~6-7× speedup, not theoretical 8×.
Worst-case (all rate-limited)~18.5s parallel vs ~148s sequential. Strictly better.

Note for future scale: if you ever raise the rerank cap above ~16 chunks, swap Promise.all for a bounded concurrency pool (5-8 in flight, queue the rest). Not needed today.

How Hermes handles concurrent LLM calls

Hermes is in-process Python on asyncio. Concurrent LLM calls are native — await asyncio.gather(*tasks) with no subprocess overhead. The framework supports a credential pool with optional per-provider rate-limiting strategies:

credential_pool_strategies  — multiple OAuth tokens, optional rate-limit policy per provider
fallback_providers  — transparent failover if primary rate-limits
api_max_retries: 3  — wraps the SDK's own retry layer

The 3-4 second CLI startup tax Atlas pays is exactly what Hermes avoids by living in the same process as its concurrency primitives. Same pattern — gather, await, partition — Atlas just spends its concurrency budget on processes instead of asyncio tasks.

05SHADOW-ATLAS GHOST WATCHDOG

A watchdog is restarting a process you turned off.

two missing flag checks · cron.ts:3441 + shadow-atlas.ts:159
15 min

What's actually happening

You disabled Shadow-Atlas after the image-vision false positives on May 19-21 by setting SHADOW_ATLAS_ENABLED=false in .env. The flag is checked in exactly one place: the IPC client at shadow-driver.ts:40-42, which short-circuits and returns {ok: false, reason: "shadow_disabled"} instead of opening the socket.

This works correctly for fireShadow() calls from relay.ts — drift checks become no-ops. But the watchdog cron at cron.ts:3441 only looks at !ping.ok. To it, "shadow_disabled" looks identical to "actual outage." Every 5 minutes it dutifully detached-spawns a fresh bun src/shadow-atlas.ts to "revive" a process you turned off.

Meanwhile, shadow-atlas.ts itself never checks the flag — its if (import.meta.main) block unconditionally calls startShadowServer(). So the PM2-managed instance is sitting there holding the named pipe, doing nothing (because no client ever connects), while the watchdog's orphan spawns hit pipe-collision EADDRINUSE and exit silently.

The real cost is lower than first thought: ~288 log lines/day plus one fast-failing Bun spawn every 5 min. The PM2 "26 restarts" figure is most likely historical noise from the pre-disable period — PM2 doesn't reset restart counts without an explicit pm2 restart, and orphan exits aren't PM2-tracked anyway. Annoying signal pollution, not a real crash loop.

How Hermes handles subsystem enable/disable

Hermes has gateway-platform enable flags directly in config.yaml, and the gateway runner orchestrates everything from one source of truth:

gateway:
  telegram:
    enabled: true
    session_idle_timeout: 1440

The gateway runner checks enabled: false in one place and skips the adapter startup, the watchdog, the scheduled jobs — all of it. One flag, every consumer reads from the same orchestrator. The Atlas bug is the structural opposite: the flag is checked in one consumer (the client lib) and missed by two others (the watchdog cron, the server itself).

The fix below isn't to adopt Hermes's whole config model — it's to apply the same principle: every component that reacts to the subsystem needs to read the same flag, not just one.
